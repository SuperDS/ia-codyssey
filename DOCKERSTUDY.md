# 🐳 Docker 실습 및 커스텀 이미지 제작 보고서

## 1. Docker 설치 및 기본 점검
Docker 환경이 정상적으로 구축되었는지 확인하고 기본 명령어를 실습했습니다.

### 1) 버전 및 데몬 확인
* **버전 확인**: `docker --version`
* **동작 여부 확인**: `docker info`
> [!TIP]
> **스크린샷 첨부**: 터미널에 출력된 버전 정보와 info 결과 이미지를 넣어주세요.

### 2) 기본 운영 명령 실습
* `docker pull ubuntu`: 도커 허브에서 우분투 베이스 이미지 다운로드
* `docker images`: 로컬에 저장된 이미지 목록 확인
* `docker ps -a`: 실행 중 및 종료된 모든 컨테이너 목록 확인

## 2. 컨테이너 실행 실습
컨테이너의 생명 주기와 격리 환경을 직접 체험했습니다.

### 1) Hello-World 실행
`docker run hello-world` 명령을 통해 이미지가 없을 때 자동으로 다운로드(Pull)하고 실행되는 과정을 관찰했습니다.

### 2) Ubuntu 컨테이너 진입
`bash
`docker run -it --name my-ubuntu ubuntu /bin/bash.


### 3) 컨테이너 종료와 유지의 차이 관찰
컨테이너는 실행 시 부여된 주 프로세스(Primary Process)의 생명주기에 따라 동작함을 확인했습니다.

* **임무 수행 후 자동 종료**: `docker run ubuntu echo "hello"`와 같이 특정 명령을 전달하면, 출력을 마친 후 컨테이너는 즉시 `Exited` 상태가 됩니다.
* **백그라운드 서비스 유지**: `-d` (detach) 옵션을 사용하여 컨테이너를 백그라운드에서 상주시키고, `docker exec` 명령어를 통해 실행 중인 컨테이너 내부에 접속하여 작업을 이어가는 방식을 학습했습니다.

## 3. 커스텀 이미지 제작 (NGINX 웹 서버)
기존에 공개된 베이스 이미지를 활용하여 나만의 정적 웹 페이지를 배포하는 커스텀 이미지를 제작했습니다.

### 1) 제작 정보
| 항목 | 내용 |
| :--- | :--- |
| **선택 베이스 이미지** | `nginx:alpine` |
| **선택 이유** | NGINX가 사전 설치되어 추가 설정이 불필요하며, Alpine 리눅스 기반으로 용량이 매우 가볍기 때문입니다. |
| **커스텀 포인트** | 로컬의 `index.html` 파일을 컨테이너 내의 웹 루트 경로(`/usr/share/nginx/html/`)로 복사하여 기본 페이지를 교체했습니다. |

### 2) Dockerfile 구성
```dockerfile
# 1. 베이스 이미지 선택
FROM nginx:alpine

# 2. 커스텀 파일 복사 (기본 페이지 교체)
COPY ./index.html /usr/share/nginx/html/

### 3) 빌드 및 실행 명령어

작성한 Dockerfile을 기반으로 이미지를 빌드하고 컨테이너를 실행하였습니다.

* **이미지 빌드**: 현재 디렉토리의 `Dockerfile`을 기반으로 `my-webserver`라는 이름의 이미지를 생성합니다.
    ```bash
    docker build -t my-webserver .
    ```
* **컨테이너 실행**: 생성된 이미지를 백그라운드에서 실행하며 포트를 연결합니다.
    ```bash
    docker run -d -p 8080:80 --name my-web-app my-webserver
    ```
* **포트 매핑 설명**: 호스트(내 컴퓨터)의 `8080` 포트로 들어오는 신호를 컨테이너 내부의 `80` 포트(NGINX 기본 포트)로 전달하도록 설정했습니다.

## 5. 학습 포인트 (정리)

이번 실습을 통해 Docker의 핵심 운용 원리를 다음과 같이 정리하였습니다.

1.  **이미지와 컨테이너의 차이**: 이미지는 실행에 필요한 모든 것이 담긴 정적인 **'설계도'**이며, 컨테이너는 이 설계도를 바탕으로 격리된 공간에서 실제 실행 중인 **'프로세스'**임을 명확히 이해했습니다.
2.  **이미지 빌드의 효율성**: `FROM` 명령어를 통해 이미 검증된 환경(NGINX 등)을 베이스로 삼고, 필요한 부분만 `COPY`하여 신속하게 배포 환경을 구축하는 Docker의 강력한 효율성을 체감했습니다.
3.  **포트 매핑의 필요성**: 컨테이너는 외부와 격리된 네트워크 환경을 가지므로, 서비스 제공을 위해서는 호스트와 컨테이너 사이의 통로인 **Port**를 연결해주는 설정이 필수적임을 알게 되었습니다.
